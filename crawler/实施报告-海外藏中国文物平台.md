# 海外藏中国文物知识管理与服务平台实施报告（爬虫与知识图谱）

## 1. 项目目标与范围

基于课程文档要求，本阶段聚焦“知识图谱构建子系统”的工程化落地，目标如下：

- 完成 3 家海外博物馆中国文物数据采集（Smithsonian / Harvard / MFA Boston）。
- 输出 UTF-8 CSV + 原图下载 + 数据质量校验。
- 支持反爬与异常重试、断点续跑、跨馆唯一标识（`museum_id + object_id`）。
- 落地双存储中的关系库写入（MySQL UPSERT）。
- 补齐知识图谱中间层：实体对齐、URI 生成、三元组导出（CSV/NT）。
- 提供增量运行台账（`crawl_runs.jsonl`）以支持审计与持续更新。

---

## 2. 当前实现架构（代码落地）

代码已模块化到 `museum_crawler/`，核心职责如下：

- `cli.py`：统一入口、参数解析、流程编排、汇总输出
- `smithsonian.py` / `harvard.py` / `mfa_boston.py`：三馆爬取实现
- `http_client.py`：会话、UA 轮换、429/5xx 退避、图片下载校验
- `text_geography.py`：朝代映射、地域抽取、文本归一化
- `io_csv.py`：批量写 CSV、与 MySQL 同批写入
- `db.py`：MySQL 自动建表（可选）+ UPSERT
- `quality.py`：关键字段完整性校验
- `kg_export.py`（新增）：三元组/实体导出与 URI 映射

## 3. 与课程要求对照（知识图谱构建部分）

### 3.1 数据爬取（已满足）

- 三馆数据源已接入，支持参数化限额与全量。
- 图片按原图策略下载，并校验有效性（Content-Type + 文件最小字节）。
- 输出 UTF-8-SIG CSV，便于 Excel 打开与后续处理。

### 3.2 反爬与稳定性（已满足）

- User-Agent 随机轮换
- 429 优先读取 `Retry-After`
- 5xx 指数退避
- 会话隔离（Smithsonian API 与图片下载分 Session）
- 分批落盘，降低中断损失

### 3.3 清洗与质量控制（基本满足）

- 字段统一化（例如多源字段合并）
- 断点续跑去重（按 `object_id`/`seen_ids`）
- 关键字段校验
- 增强：跨馆主键升级为 `museum_id + object_id`，避免 ID 冲突

### 3.4 数据建模（三元组）（新增后满足）

新增 `kg_export.py`，自动从 CSV 构建：

- `kg_entities.csv`：实体清单（URI、类型、标签、归一化标签、来源计数）
- `kg_triples.csv`：三元组（`subject/predicate/object/object_type`）
- `kg_artifact_uri_map.csv`：业务主键到 URI 映射
- `kg_triples.nt`：N-Triples 文件（便于导入图数据库）

覆盖核心实体：Artifact / Museum / Dynasty / Artist / Location。  
覆盖核心关系：`collectedBy`、`locatedIn`、`belongsToDynasty`、`createdBy` 等。

### 3.5 实体对齐与 URI（新增后满足“可用版本”）

- 采用归一化规则（小写、空白压缩、标点清洗）做软对齐键。
- 艺术家支持多分隔符拆分（`; , 、 and & /`）。
- URI 统一前缀（默认可配置）：`--kg-base-uri`。
- 产出可追溯映射文件，支持后续人工审核/纠偏。

### 3.6 双层存储（阶段性满足）

- 已落地 MySQL。
- 图数据库（Neo4j/Virtuoso）尚未直接写入，但已输出标准中间件文件（CSV/NT），
  可直接用于下一阶段导入与 Cypher/SPARQL 验证。

### 3.7 增量更新（部分满足）

- 当前支持“基于已有 CSV 的断点续跑去重”。
- 新增 `crawl_runs.jsonl` 运行台账，记录参数、产量、KG 统计，用于增量审计。
- 下一步建议补充“源侧更新时间字段驱动的真正增量策略”。

## 4. 关键技术方案说明

### 4.1 跨馆唯一标识

- 方案：`museum_id`（1/2/3）+ `object_id`
- 优点：兼容馆内原始编号，不破坏来源可追溯性；跨馆合并无主键冲突。

### 4.2 失败可恢复

- 每 10 条批量写盘 + 异常兜底 flush
- 断点加载已爬 ID
- 网络/限流可重试

### 4.3 朝代映射准确性

- `period -> dynasty` 使用词边界匹配，解决 `Qing` 被误判为 `Qin` 的问题。
- 长词优先（`Northern Song` 优先于 `Song`）。

### 4.4 数据可追溯

- 保留 `detail_url`、`image_url`、`crawl_date`
- 运行级台账：`crawl_runs.jsonl`
- 三元组导出保留 artifact URI 与来源映射

## 7. 大厂标准分工方案（6 人配置）

### 角色与职责

1. **采集引擎负责人（1人）**
   - 负责三馆爬虫稳定性、反爬策略、重试退避、性能优化
   - 交付：可持续运行的采集流水线、错误分级策略

2. **数据治理工程师（1人）**
   - 负责字段标准化、去重、质量规则、异常数据闭环
   - 交付：质量规则库、清洗报表、空值/异常修复策略

3. **知识图谱工程师（1人）**
   - 负责实体建模、关系建模、URI 规范、实体对齐策略
   - 交付：三元组导出规范、实体映射字典、对齐评估结果

4. **后端与存储工程师（1人）**
   - 负责 MySQL/图数据库写入、索引与查询接口
   - 交付：数据服务 API、写入幂等策略、备份恢复流程

5. **测试与SRE（1人）**
   - 负责自动化测试、监控告警、运行台账、故障演练
   - 交付：测试用例集、运行看板、值班手册

6. **产品/项目经理（1人，可由组长兼任）**
   - 负责需求拆解、里程碑排期、验收口径、跨端联调
   - 交付：迭代计划、风险清单、阶段评审材料

### 迭代节奏（建议）

- **第1周**：爬虫稳定 + CSV/图片落地 + MySQL 基础入库
- **第2周**：KG 导出 + 对齐规则 + 图数据库导入验证
- **第3周**：Web 查询与可视化联调 + QA 回归 + 性能压测
- **第4周**：问答/RAG 接口联调 + 文档沉淀 + 最终答辩材料

## 8. 风险与改进计划

### 主要风险

- 外部 API/站点限流与策略变更
- 源数据字段不稳定（页面结构调整）
- 同名实体误对齐（艺术家、地点）

### 已有缓解

- 重试退避 + 日志细分 + 分批持久化
- 多策略提取（API摘要 + content补全）
- 对齐采用规范化键 + 可追溯映射文件

### 下一步建议（高优先）

- 接入 Neo4j 导入脚本（CSV/NT -> Cypher）
- 增加实体对齐“人工复核白名单/黑名单”
- 增加单元测试与回归样本集（固定 50 条基准样本）
- 增加 Docker 化运行与一键部署
